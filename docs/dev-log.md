目的

実装
    今回は、JavaScriptを使ってタイマー機能を実装した。

    最初にHTMLでタイマーの表示部分とボタンを用意し、その後JavaScriptで動きをつける形で進めた。

発生したバグ
    「Start」ボタンを押してもカウントアップが始まらないバグに遭遇。

原因
    『時間表示を更新する関数』に含まれる「timeDisplay.textContent =」から下のコードで、小文字と大文字の打ち間違い。（string、padstart）

解決方法
    それぞれの文字を大文字に打ち直す。
        ・string → String
        ・padstart → padStart

学んだこと

    ・エラー発生時に、ブラウザのコンソール（F12）を確認することで、原因特定が効率的に行なえることを実体験として学んだ。

    ・setInterval() を使うことで、一定時間ごとに処理を繰り返せることを理解したが、
    どのタイミングで止めるか（clearInterval）についてはまだ理解が浅いと感じた。

    ・秒数をそのまま表示するのではなく、計算して「分:秒」に変換している処理について、仕組みはなんとなく理解できるが、自分でゼロから書けるレベルではないと感じた。

    ・コードは一気に理解しようとすると難しいが、1行ずつ動作を確認しながら進めることで動くものが作れることがわかった。

感想

## 2026/03/21

実装
    タイマーの精度問題を改善するため、カウントダウン処理を「終了時刻ベース」に変更した。

背景・課題
    従来の実装では、setInterval によって1秒ごとに seconds を減算していたが、
        ・タブ非アクティブ時に処理が遅延する
        ・リロード時に残り時間がズレる
        ・setInterval の実行タイミングに誤差がある
    といった問題が発生していた。

対応内容
    ①終了時刻ベースへの変更
        ・endTime を新たに導入
        ・スタート時に「現在時刻 + 残り秒数」で終了時刻を設定
            endTime = Date.now() + totalSeconds * 1000;

        ・カウントダウン時は、毎回現在時刻との差分から残り時間を算出
            const diff = endTime - Date.now();
            const remaining = Math.ceil(diff / 1000);
            seconds = remaining;
    
    ②終了判定の修正
        ・秒ではなくミリ秒ベースで判定するように変更
            if (diff <= 0)

    ③ストップ処理の修正
        ・停止時に残り時間を再計算し、seconds に保持
        ・endTime をリセットすることで再開可能にした
    
発生したバグと対応
    ①カウントダウンが表示されない
        原因：remaining を seconds に代入していなかった

        対応：seconds = remaining;
    
    ②初回1秒が反映されない
        原因：setInterval が1秒後に実行されるため

        対応：countDown(); をStart直後に1回実行
    
    ③表示ズレ（2秒飛び）
        原因：Math.floor による切り捨て

        対応：Math.ceil() に変更
    
    ④00:01 で停止する問題
        原因：秒ベースで終了判定していた

        対応：diff <= 0 に変更
    
    ⑤ alert による表示ズレ
        原因：alert がメインスレッドをブロックするため、表示更新が遅れる

        対応：
            setTimeout(() => {
                console.log("時間終了！");
            }, 0);

学び
    ・setInterval は“正確な時間管理”には向かない
    ・時間処理は「状態更新」ではなく「差分計算」で行なうべき
    ・表示用の値（seconds）と、基準となる時間（endTime）は分けて考える必要がある
    ・バグの多くは「設計の混在」によって発生する（旧ロジック ＋ 新ロジック）

今後の改善
    ・アラーム停止ボタンの追加
    ・UIの視認性向上
    ・alert の代替（モーダル表示 など）

        ・コードの見通しが良くなる
        ・バグの原因が特定しやすくなる

        変数をまとめるだけでここまでわかりやすくなるとは思っていなかった。
        今後は最初から state で設計することを意識したい。

実装②　setTimeout（tick関数）への変更

    原因
        setInterval を使っていたため、
        ・処理が遅れたときにズレが発生する
        ・停止処理がわかりにくい
        問題があった。
    
    解決方法
        setTimeout を使った再帰処理に変更した。
            function tick() {
                countDown();
                state.timerId = setTimeout(tick, 1000);

    学んだこと
        ・setInterval は「一定時間ごとに無条件で実行」
        ・setTimeout は「処理後に次を予約する」
        という違いがあることを理解した。
    
    感想

実装③　バグ①：タイマーが止まらない（アラームが鳴り続ける）

    原因

    解決方法
        tick関数に停止条件を追加した。
            if (state.remainingSeconds <= 0) return;
        
    学んだこと
        非同期処理では「どこで止めるか」を明確にしないと、処理が止まらなくなることを理解した。
    
    感想
        見た目ではわかりづらいバグだったが、原因はシンプルだった。
        非同期処理は特に注意が必要と感じたバグだった。

実装④　バグ②：タイマーが動かない

    原因
        tick関数の先頭で
            if (!state.isRunning) return;
        としているのに、

        start処理で、

            state.isRunning = true;
        という順番になっていたため、

        最初の tick が実行されず終了していた。
    
    解決方法
        順番を以下の記述に修正した。
            state.isRunning = true;
            tick();

    学んだこと
        処理の「順番」が非常に重要であることを理解した。
        特に非同期処理では、順番がズレると全く動かなくなる。

## 2026/04/04

発覚したバグ
    タイマー終了後に別の時間を設定して再度開始した際、前回の時間でカウントダウンが開始されてしまう不具合が発生した。
    なお、ページ再読み込み（F5）すると正常に動作する状態だった。

原因
    タイマー終了後も、state.totalSeconds に前回の時間が保持されたままになっていたため、再スタート時の分岐処理において新しい入力値ではなく既存の state が優先されていた。
    具体的には、以下の条件が成立してしまうことで、新規入力の処理に入らず前回の時間が再利用されていた。

        else if (state.totalSeconds > 0)
    
    また、タイマー終了時に remainingSeconds を totalSeconds に戻していたため、状態としては「未初期化のまま再利用可能な状態」になっていた。

解決方法
    タイマー終了時に state を初期化する処理を追加し、次回のスタート時に必ず新規入力が使われるように修正した。

        state.remainingSeconds = 0;
        state.totalSeconds = 0;
    
    これにより、再スタート時に state.totalSeconds > 0 の条件が成立しなくなり、新しい入力値をもとにタイマーが開始されるようになった。

学んだこと
    ・バグの原因はロジックだけでなく「状態の残り方」にもある
    ・特に totalSeconds のような基準値は、不要になった時点で明示的にリセットする必要がある
    ・「終了」「停止」「リセット」で state の扱いを整理しないと、状態の不整合が発生しやすい
    ・条件分岐を増やす前に、「state を正しく管理できているか」を確認することが重要

感想
    今回のバグは一見すると分岐処理の問題に見えたが、実際には state の管理が不十分だったことが原因だった。
    これまで「処理の流れ」に注目することが多かったが、「状態がどう変化しているか」を意識する重要性を理解できた。

    また、F5で解決する、という現象から「初期化の問題ではないか」と考えられた点は、原因特定のヒントとして有効だった。

    今後は同様のケースにおいて、状態の初期化タイミングを優先的に疑うようにしたい。

## 2026/04/05

発覚したバグ
    タイマーを開始後、途中で Stop → Reset を実行し、その後別の時間を設定して再度Startした際、本来は新しい設定時間で開始されるべきところ、前回中断したタイマーが再利用されてカウントダウンが開始されるバグを確認した。

原因
    Reset処理において、state.remainingSeconds や endTime は初期化されていたが、**タイマーの元データである state.totalSeconds がリセットされていなかった**。

    そのため、Start処理内の以下の分岐により、過去の設定時間が再利用されていた。

        else if (state.totalSeconds > 0) {
            state.remainingSeconds = state.totalSeconds;
        }

解決方法
    Reset時にタイマーの状態を完全に初期化するため、stateを一元的にリセットする resetState 関数を作成し、Reset処理内で呼び出すように修正した。

    これにより、totalSeconds を含むすべての状態がリセットされ、過去の設定が再利用される問題を解消した。

学んだこと
    ・状態管理においては、**一部の値だけをリセットすると不整合が発生する**
    ・totalSeconds のような「ロジックの基準になる値」が残っていると、意図しない挙動につながる
    ・Reset処理は「見た目」ではなく「内部処理」を完全に初期化する必要がある
    ・状態初期化処理を関数にまとめることで、**バグの再発防止と可読性の向上に繋がる**

## 2026/04/07

発覚したバグ①
    プログレスバーが表示されない

原因
    HTMLやJavaScriptの処理不足
        progressBarのDOM取得 や width更新処理が不足していた

解決方法
    progressBarのHTML/CSS/JSをすべて実装
        DOM取得 + width更新処理を追加

発覚したバグ②
    Resume時にプログレスバーがリセットされる

原因
    totalSeconds を remainingSeconds で上書きしていた
        進捗の基準が変わり、常に0％扱いになってしまう

解決方法
    totalSeconds を変更しないように修正
        基準地として固定

発覚したバグ③
    Resume時にカウントダウンが最初から始まる

原因
    endTimeの計算が固定だった
        常に「初期時間」で再計算されていた

解決方法
    endTime を remainingSeconds ベースに変更
        state.endTime = Date.now() + state.remainingSeconds * 1000;

発覚したバグ④
    タイマー終了時にプログレスバーが0に戻る

原因
    終了時に totalSeconds を0にしていた
        renderProgressで強制的に0％にリセットされていた

解決方法
    終了時に totalSeconds を保持
        renderProgress で100％表示に固定

その他の処理
    renderProgress を updateDisplay に統一
    → 描画処理を一ヶ所に集約
    → バグを発生しにくくする設計

学んだこと
    ・バグの多くは「状態の扱いミス」で起きる
    ・totalSeconds と remainingSeconds の役割を分けることが重要
    ・UIの不具合は計算ロジックから確認するべき
    ・処理は“定義する”だけでなく“呼び出す”ことが必要
    ・描画処理を分散させると不具合の原因になる

    ・見た目の問題でも、原因はロジックにあることが多い
    ・修正は「1箇所ずつ原因を切り分ける」ことが重要

## 2026/04/08

目的
    タイマーアプリのUI改善として、スマホでも操作しやすいレスポンシブデザインを実装する。

実装内容

    ①レイアウト調整（Flexbox導入）
        ・.inputs と .buttons に display: flex を適用
        ・PCでは横並びレイアウトに設定

            .inputs {
                display: flex;
            }
            
            .buttons {
                display: flex;
            }

    ②レスポンシブ対応
        画面横幅480px以下で縦並びに変更
            @media (max-width: 480px) {
                .inputs {
                    flex-direction: column;
                }

                .button {
                    flex-direction: column;
                }
            }
    
    ③中央配置の調整
        スマホ時に中央寄せを実装
            align-items: center;

    ④タップしやすいUIに改善
        ・ボタン・入力欄のサイズを調整
        ・高さ44px以上を確保
    
発生した問題
    ①スマホサイズでも縦並びにならない
    ②中央寄せにならない
    ③要素の横幅が広がりすぎる
    ④ボタン・入力欄が小さく操作しづらい

原因
    ①CSSクラス名の不一致
        ・ .input / .inputs
        ・ .button / .buttons

    ②display:flex が適用されていない状態で flex-direction を使用していた

    ③flex-direction による字句の変化を理解していなかった
        ・column時に justify-content は縦方向に作用する
    
    ④flexのデフォルト挙動（要素が広がる）を考慮していなかった

解決方法
    ①クラス名をHTMLとCSSで統一

    ②display:flex を前提としてレイアウトを再構築
    
    ③中央寄せを以下で修正
        align-items: center;

    ④横幅制御を追加
        max-width: 100%;
    
    ⑤UIサイズ調整
        height: 44px;
    
学んだこと
    ・flex は「display:flex」が前提で動く
    ・flex-direction によって軸が変わる
    ・justify-content と align-items は役割が異なる
    ・CSSが効かない原因はセレクタミスが多い
    ・スマホUIでは「タップしやすさ」が重要

感想
    レスポンシブ対応は単に見た目を変える作業ではなく、「ユーザーが使いやすいUIを設計すること」だと理解した。

    特に、『flex の軸の考え方』と『クラス名のミス』は今後も発生しやすいため、開発時には意識して確認する必要があると感じた。
